# LangGraph 核心知识整理

## 1. 什么是 LangGraph

- **定位**：低级编排框架，用于构建、管理和部署**有状态的长期运行 Agent**。
- **核心建模**：将 Agent 工作流视为**图（Graph）**。
  - **节点（Nodes）**：计算单元（LLM 调用、工具执行、自定义逻辑）。
  - **边（Edges）**：定义节点间的转换逻辑（决定执行流程）。
  - **状态（State）**：在整个图执行中共享和传递的数据。
- **关键能力**：持久化执行、人机协作（中断/恢复）、记忆管理（短期/长期）、流式处理、生产级部署。

## 2. 与 LangChain 的区别

| 特性     | LangGraph                  | LangChain    |
| :------- | :------------------------- | :----------- |
| 抽象级别 | 低，细粒度控制             | 高，开箱即用 |
| 状态管理 | 内置状态机 + Checkpointer  | 需自行管理   |
| 执行模型 | 基于图的并行执行（Pregel） | 线性链式     |
| 持久化   | 原生支持                   | 需额外实现   |
| 适用场景 | 复杂、有状态、多智能体     | 简单链式调用 |

------

## 3. 快速入门流程（以 RAG + 搜索为例）

1. **定义状态**（`TypedDict` 或 `BaseModel`）。
2. **定义节点函数**（接收 state，返回增量更新）。
3. **构建 `StateGraph`** → 添加节点、添加边（普通边/条件边）。
4. **编译图** → 得到可调用的 `compiled_graph`。
5. **调用**（`invoke` / `stream`）。

> 示例结构：`START` → `rag_search_node` & `web_search_node`（并行）→ `final_answer_node` → `END`。

------

## 4. 状态（State）深度解析

### 4.1 状态定义方式

- **TypedDict**（推荐）、**Pydantic BaseModel**、**Dataclass**。
- 支持**输入/输出隔离**：
  - `state_schema`：完整内部状态（必填）。
  - `input_schema`：允许的输入字段（子集）。
  - `output_schema`：输出的字段（子集）。
- **节点间数据隔离**：可为不同节点指定不同状态子集（私有状态）。

### 4.2 Reducer 函数（合并增量更新）

- **默认**：覆盖更新。
- **内置**：`operator.add`（列表追加、字符串连接、数值累加）。
- **自定义**：接收 `(left, right)` 返回合并值。
- 用法：在状态字段上用 `Annotated[Type, reducer_func]` 标记。

### 4.3 状态的存储与持久化

- **Checkpointer**（检查点机制）：
  - 存储每个超步后的状态快照。
  - 实现类：`InMemorySaver`（内存）、`SqliteSaver`、`RedisSaver` 等。
- **thread_id**：隔离不同会话/用户的上下文空间。
  - 同一 `thread_id` 多次调用保持状态（短期记忆）。
  - 新 `thread_id` 开启新会话。

### 4.4 故障恢复（断点续传）

- 编译图时传入 `checkpointer`。
- 调用时传入 `config`（包含 `thread_id`）。
- 节点失败后，修复代码，**再次 `invoke(None, config=config)`** 即可从最后检查点继续执行（节点重入需保证幂等性）。

### 4.5 历史状态获取

- `graph.get_state(config)` → 最近一次状态快照。
- `graph.get_state_history(config)` → 所有历史快照（倒序）。
- 快照包含：`values`、`next`（待执行节点）、`metadata`、`interrupts` 等。

## 5. 节点（Nodes）

### 5.1 节点输入/输出

- **输入**：`state`（必须）、`config`（`RunnableConfig`）、`runtime`（`Runtime` 含 `context` 和 `stream_writer`）。
- **输出**：**必须只返回增量更新**（不能返回整个 state），否则会导致合并异常或数据错乱。

### 5.2 特殊节点

- **START**：虚拟起点，将用户输入注入图。
- **END**：虚拟终点，表示终止。

### 5.3 节点缓存

- 编译时指定 `cache`（如 `InMemoryCache`、`RedisCache`）。
- 添加节点时配置 `CachePolicy(ttl=秒, key_func=...)`。
- 相同输入直接返回缓存结果。

### 5.4 节点重试

- 通过 `retry_policy=RetryPolicy(max_attempts=N, retry_on=(异常类型,))` 配置。
- 默认不重试的异常：`ValueError`、`TypeError`、`ImportError` 等。

### 5.5 图内外数据传递（流式输出）

- `graph.stream(stream_mode=...)` 支持多种模式：
  - `"values"`：每个超步后完整状态。
  - `"updates"`：每个节点的增量更新。
  - `"custom"`：节点内通过 `runtime.stream_writer` 发送的自定义数据。
  - `"messages"`：LLM 的 token 流（与消息元组）。
  - `"debug"`：所有调试信息。
  - 混合模式：传入列表同时获取多种数据。

### 5.6 人工审核（interrupt）

- 在节点内部调用 `interrupt(payload)`。
- 图会暂停，返回 `payload` 给调用方，同时生成检查点。
- 恢复时使用 `Command(resume=user_input)` 继续执行。
- **注意**：`interrupt` 所在函数在恢复时会从头执行，需确保幂等性（如将可能重复执行的操作移出该函数）。

------

## 6. 边（Edges）

### 6.1 普通边与条件边

- **普通边**：直接从一个节点到另一个节点（`add_edge`）。
- **条件边**：通过路由函数动态决定下一个节点（`add_conditional_edges`）。
  - 路由函数接收 `state`，返回目标节点名（或列表）。
  - 需提供 `path_map` 映射返回值到实际节点名。

### 6.2 可控循环

- 利用条件边形成循环（如 ReAct 模式）。
- 必须设置终止条件，否则会无限循环。
- 可通过 `config={"recursion_limit": N}` 限制最大超步数，超出抛出 `GraphRecursionError`。

------

## 7. 底层执行模型 —— Pregel

- LangGraph 底层基于 **Pregel 模型**（BSP 式并行计算）。
- **超步（SuperStep）** 三阶段：
  1. **Plan**：确定本步要执行的 Actors（节点）。
  2. **Execute**：并行执行所有选定 Actors。
  3. **Update**：用 Actors 的输出更新 Channels（通道）。
- **Channels**：节点间的通信通道，每个通道有值类型和更新函数（Reducer）。
- **Actors（PregelNode）**：订阅和写入 Channels，实现节点逻辑。

> 图中每个节点相当于一个 Actor，边定义了订阅/写入的 Channel 关系。

------

## 8. 实践关键点总结

- 状态设计优先使用 `TypedDict` + `Annotated` 标记 Reducer。
- 生产环境必须使用持久化 Checkpointer（如 Sqlite/Redis）。
- 使用 `thread_id` 隔离会话，利用 Checkpointer 实现多轮对话和故障恢复。
- 流式输出根据前端需求选择合适模式（常用 `updates` + `custom`）。
- 人工审核节点合理使用 `interrupt`，注意幂等性。
- 循环图务必设置终止条件和递归限制。
- 缓存和重试策略用于提升性能和稳定性
