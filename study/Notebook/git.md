

## git命令汇总

```bash
一、全局配置命令
# 设置提交用户名
git config --global user.name "名称"
# 设置提交邮箱
git config --global user.email "邮箱"
# 查看全部配置
git config --list

二、本地仓库初始化与状态查看
# 初始化本地git仓库
git init
# 查看工作区、暂存区状态
git status

三、提交流程（工作区→暂存区→本地库）
# 单个文件加入暂存区
git add 文件名
# 全部修改文件加入暂存区
git add .
# 暂存区提交到本地仓库，带备注
git commit -m "提交说明"

四、版本历史与版本回退
# 简洁查看所有提交版本（含版本号）
git reflog
# 查看完整提交日志
git log
# 回退到指定历史版本
git reset --hard 版本号

五、分支全套命令
# 创建新分支
git branch 分支名
# 查看所有分支
git branch -v
# 切换到指定分支
git checkout 分支名
# 将目标分支合并到当前分支
git merge 分支名
分支冲突专用命令
# 冲突文件修改完成后加入暂存区
git add 冲突文件名
# 冲突解决提交，不加文件名称
git commit -m "解决冲突备注"

六、远程仓库操作（Gitee/GitHub）
# 查看已绑定远程仓库别名与地址
git remote -v
# 绑定远程仓库，别名默认origin
git remote add origin 远程仓库HTTPS/SSH地址
# 推送本地分支到远程对应分支
git push origin 本地分支名
# 拉取远程最新代码并自动合并本地
git pull origin 远程分支名
# 克隆远程完整仓库到本地
git clone 远程仓库地址

七、SSH免密配置命令
# 生成SSH密钥对
ssh-keygen -t ed25519 -C "注册邮箱"
# 读取公钥内容，复制到Gitee部署公钥
cat ~/.ssh/id_ed25519.pub
# 测试SSH连通性
ssh -T git@shturl.cc
```

## 第1章 Git概述

### 1. 什么是版本控制
- **定义**：一种记录文件内容变化，以便将来查阅特定版本修订情况的系统。
- **核心价值**：记录文件修改历史，支持查看历史版本，方便版本切换和回退。

### 2. 为什么需要版本控制
- **个人开发**：可以管理自己的代码迭代，随时回溯。
- **团队协作**：多人并行开发时，版本控制能解决代码冲突、合并变更，是协同工作的基础。

### 3. 版本控制工具分类

| 类型       | 代表工具                      | 特点                                                         |
| ---------- | ----------------------------- | ------------------------------------------------------------ |
| **集中式** | CVS、SVN（Subversion）、VSS   | 单一中央服务器保存所有版本，客户端联网操作。优点是权限管理方便；缺点是**单点故障**——服务器宕机则无法提交或协同。 |
| **分布式** | Git、Mercurial、Bazaar、Darcs | 每个客户端都**完整镜像整个仓库**（包含历史）。优势：断网可本地开发，安全性高（任何客户端都可恢复数据）。 |

### 4. Git 简介
- **定位**：免费、开源的**分布式版本控制系统**，适用于从小型到大型的各种项目。
- **特点**：学习曲线平缓，占用空间小，性能极快。
- **核心优势**：拥有廉价的本地库、便捷的暂存区域，以及强大的分支与工作流支持。
- **性能对比**：普遍优于 SVN、CVS、Perforce、ClearCase 等传统工具。

### 5. Git 简史
- （原文图片未显示，此处补充关键背景）Git 由 Linux 之父 Linus Torvalds 于2005年创建，最初是为了更好地管理 Linux 内核开发，现已发展为全球最流行的版本控制系统。

### 6. Git 工作机制（核心流程）
虽然原文只有一张图，但实际 Git 的工作流程分为四个主要区域，理解它们至关重要：

1. **工作区（Workspace）**：你电脑上实际编写、修改文件的目录。
2. **暂存区（Staging Area / Index）**：临时存放你准备提交的改动，可以理解为“提交前的缓冲区”。
3. **本地仓库（Local Repository）**：存放在你本地计算机上的版本库，包含所有提交的历史记录。
4. **远程仓库（Remote Repository）**：托管在服务器上的共享仓库，用于团队同步（如 GitHub、GitLab、Gitee）。

**基本工作流**：工作区修改 → `git add` → 暂存区 → `git commit` → 本地仓库 → `git push` → 远程仓库。

### 7. Git 与代码托管中心（远程库）
- **定义**：基于网络服务器的远程代码仓库，用于团队协作和备份。
- **常见平台**：
  - **局域网**：GitLab（企业自建）
  - **互联网**：GitHub（国外）、Gitee 码云（国内，访问速度快）

| `git log`                              | 查看历史记录（详细版）   |

### 1. 设置用户签名

**作用**：标识操作者身份，每次提交都会记录该信息，便于追踪是谁做的提交。**首次安装Git必须设置**，否则无法提交。

**语法**：
```bash
git config --global user.name 用户名
git config --global user.email 邮箱
```

**案例**：
```bash
git config --global user.name Layne
git config --global user.email Layne@atguigu.com
```

**查看配置**：
```bash
git config --list          # 查看所有全局配置
cat ~/.gitconfig           # 直接查看配置文件内容（Linux命令）
```

> **注意**：这里的用户名和邮箱仅用于本地标识，**与GitHub/Gitee等远程仓库的登录账号无关**。

---

### 2. 初始化本地库

```bash
git init
```

**案例**：在项目根目录执行后，会生成一个隐藏的 `.git` 文件夹，该目录即为本地仓库。

---

### 3. 查看本地库状态

```bash
git status
```

**案例场景**：
- **首次查看（工作区无文件）**：提示当前分支（如 `master`），并显示“没有提交”或“nothing to commit”。
- **新增文件后查看**：会显示未追踪的文件列表（红色），提示“Untracked files”。
- **修改文件后查看**：会显示已修改但未暂存的文件（红色），提示“Changes not staged for commit”。

---

### 4. 添加暂存区

```bash
git add 文件名
```

**案例**：将工作区的 `hello.txt` 添加到暂存区后，再执行 `git status`，该文件会显示为绿色，状态变为“Changes to be committed”，表示已进入暂存区。

---

### 5. 提交本地库

```bash
git commit -m "日志信息" 文件名
```

**案例**：提交后，`git status` 会提示“nothing to commit, working tree clean”，表示工作区和暂存区干净，提交成功。

> **日志信息**：建议写清晰、有意义的描述，便于后续查看历史记录时理解每次提交的目的。

---

### 6. 修改文件后的完整流程

当已提交的文件被修改后，需要重复“添加→提交”两步：

1. **查看状态**：`git status` 会显示“Changes not staged for commit”（工作区有修改）。
2. **添加到暂存区**：`git add 文件名`。
3. **再次查看状态**：文件变为绿色，状态为“Changes to be committed”。
4. **提交到本地库**：`git commit -m "新的日志" 文件名`。

---

### 7. 历史版本管理

##### 7.1 查看历史版本
- **精简版**：`git reflog` —— 显示简短的版本号（前几位）和提交信息，适合快速浏览。
- **详细版**：`git log` —— 显示完整的提交哈希、作者、日期和完整日志信息。

**案例输出示例**（通过 `git reflog`）：
```
48f4e22 (HEAD -> master) HEAD@{0}: commit: 第二次提交：修改hello.txt
8ca80d7 HEAD@{1}: commit (initial): 第一次提交：创建hello.txt
```

##### 7.2 版本穿梭（切换版本）

```bash
git reset --hard 版本号
```

**案例步骤**：
1. 先查看历史记录，确认当前版本（例如当前在 `48f4e22`）。
2. 执行切换命令，回到之前的版本（例如 `8ca80d7`）：
   ```bash
   git reset --hard 8ca80d7
   ```
3. 切换后再次查看历史记录，会发现 `HEAD` 指向了目标版本。
4. 查看文件内容，会发现文件已恢复为对应版本的内容。

**核心原理**：Git切换版本时，实际是**移动 HEAD 指针**，指向不同的提交对象，因此速度极快。

---

### 补充小贴士
- 所有命令都是在 **Git Bash** 终端中执行。
- 文件名支持通配符（如 `git add .` 添加所有变更），但本笔记基础阶段建议明确指定文件名。
- `git reflog` 记录了所有 HEAD 的移动历史，即使切换回旧版本后，也能找到新版本的哈希值，方便再切回来。

## 第4章 Git分支操作

### 1. 什么是分支

在版本控制过程中，当需要同时推进多个任务时，可以为每个任务创建独立的分支。使用分支意味着程序员可以把自己的工作从开发主线上分离开来，开发自己分支的时候，**不会影响主线分支的运行**。

对于初学者而言，分支可以简单理解为"副本"——一个分支就是一个单独的副本。

> **底层原理**：分支本质上是指针的引用，指向某个具体的提交记录。

---

### 2. 分支的好处

| 好处         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **并行开发** | 同时并行推进多个功能开发，提高开发效率                       |
| **隔离风险** | 各个分支独立开发，如果某一个分支开发失败，不会对其他分支有任何影响，失败的分支删除重新开始即可 |

---

### 3. 分支操作命令速查

| 命令                  | 作用                     |
| --------------------- | ------------------------ |
| `git branch 分支名`   | 创建分支                 |
| `git branch -v`       | 查看所有分支             |
| `git checkout 分支名` | 切换分支                 |
| `git merge 分支名`    | 把指定分支合并到当前分支 |

---

#### 3.1 查看分支

```bash
git branch -v
```

**示例**：执行后会列出所有本地分支，当前所在分支前会有 `*` 标记。

---

#### 3.2 创建分支

```bash
git branch 分支名
```

**示例**：创建一个名为 `hot-fix` 的分支。
```bash
git branch hot-fix
```

---

#### 3.3 在分支上修改文件（以master分支为例）

1. 在 `master` 分支上修改 `hello.txt` 文件内容。
2. 添加到暂存区：`git add hello.txt`
3. 提交到本地库：`git commit -m "master分支修改" hello.txt`
4. 查看分支：`git branch -v` 确认所有分支指向的版本。

---

#### 3.4 切换分支

```bash
git checkout 分支名
```

**示例**：从 `master` 切换到 `hot-fix` 分支。
```bash
git checkout hot-fix
```

切换后观察文件内容，会发现与 `master` 分支的内容不同——因为不同分支的修改互不影响。

**在hot-fix分支上修改的完整流程**：
1. 修改 `hello.txt` 文件内容。
2. 添加到暂存区：`git add hello.txt`
3. 提交到本地库：`git commit -m "hot-fix分支修改" hello.txt`

---

#### 3.5 合并分支

```bash
git merge 分支名
```

> **注意**：该命令是将 **指定分支** 合并到 **当前所在分支**。

**示例**：将 `hot-fix` 分支合并到 `master` 分支。
1. 先切换回 `master` 分支：`git checkout master`
2. 执行合并：`git merge hot-fix`

---

#### 3.6 产生冲突

**冲突表现**：执行 `git status` 时，状态后面会显示 `MERGING`。

**冲突原因**：两个分支在 **同一个文件的同一个位置** 有两套完全不同的修改，Git无法替我们决定使用哪一个，必须**人为决定**新代码内容。

**查看状态**：`git status` 会检测到文件有两处冲突修改，提示 `both modified`。

---

#### 3.7 解决冲突

**解决步骤**：

**第一步：编辑有冲突的文件**
- 打开冲突文件，找到特殊标记：
  ```
  <<<<<<< HEAD
  当前分支（如master）的代码内容
  =======
  被合并分支（如hot-fix）的代码内容
  >>>>>>> hot-fix
  ```
- **人为决定**保留哪些内容，删除特殊符号，保留最终需要的代码。

**第二步：添加到暂存区**
```bash
git add 文件名
```

**第三步：提交（注意：不能带文件名）**
```bash
git commit -m "日志信息"
```
> ⚠️ 解决冲突后的 `git commit` **不能带文件名**，否则会报错。

完成后，`MERGING` 状态消失，恢复正常。

---

### 4. 创建分支和切换分支图解（原理说明）

用文字说明核心原理：

- **分支本质**：`master`、`hot-fix` 等分支名，其实都是指向具体版本记录的**指针**。
- **HEAD指针**：`HEAD` 是一个特殊指针，它指向**当前所在的分支**。`HEAD` 指向 `master`，就在 `master` 分支；指向 `hot-fix`，就在 `hot-fix` 分支。
- **创建分支**：本质就是**多创建一个指针**，指向某个提交记录。
- **切换分支**：本质就是**移动 HEAD 指针**，指向另一个分支。

> 这种基于指针的设计，让Git的分支操作极其轻量和高效。

---

#### 分支操作完整流程图解（文字版）

```
提交记录：  [A] --- [B] --- [C]  ← master (HEAD)
                              ↑
                           [hot-fix]
```

- 在 `C` 版本处创建 `hot-fix` 分支，两个分支都指向 `C`。
- 在 `hot-fix` 上继续开发，提交到 `D`，而 `master` 仍指向 `C`。
- 切换分支就是移动 `HEAD` 指针，指向目标分支。

## 第5章 Gitee（码云）操作

### 1. Git代码托管服务概述

Git本地仓库和远程仓库之间需要借助代码托管平台进行同步。常用的代码托管服务有：

| 平台              | 地址                      | 特点                                               |
| ----------------- | ------------------------- | -------------------------------------------------- |
| **GitHub**        | https://github.com/       | 全球最大的开源托管平台，服务器在国外，速度相对较慢 |
| **码云（Gitee）** | https://gitee.com/        | **国内**平台，服务器在国内，速度快，适合国内开发者 |
| **GitLab**        | https://about.gitlab.com/ | 开源项目，支持企业自建部署，常用于企业内部代码管理 |

---

### 2. Gitee简介

- **是什么**：一个基于Git的代码托管平台，提供版本托管服务。
- **能干嘛**：Git本身没有中央服务器，需要将代码放到远程服务器上才能与他人协作。Gitee就是这样的远程仓库托管平台。
- **去哪下/注册**：https://gitee.com/

---

### 3. 码云账号注册和登录

1. 进入官网 https://gitee.com/ ，点击右上角"注册Gitee"。
2. 填写个人信息完成注册。
3. 注册成功后直接登录，进入码云首页。

---

### 4. 创建远程仓库

1. 登录后，点击右上角 **"+"** 号，选择"新建仓库"。
2. 填写仓库信息（仓库名称、路径、简介等）。
3. 选择仓库是否公开（公开/私有）。
4. 点击"创建"按钮，仓库创建成功。
5. 创建完成后，会生成远程仓库地址（HTTPS和SSH两种格式）。

---

### 5. 远程仓库操作命令速查

| 命令                             | 作用                             |
| -------------------------------- | -------------------------------- |
| `git remote -v`                  | 查看当前所有远程地址别名         |
| `git remote add 别名 远程地址`   | 为远程仓库地址起别名             |
| `git push 别名 分支名`           | 推送本地分支内容到远程仓库       |
| `git clone 远程地址`             | 将远程仓库克隆到本地             |
| `git pull 远程库别名 远程分支名` | 拉取远程最新内容并与本地分支合并 |

---

#### 5.1 查看远程地址别名 & 添加别名

**查看所有远程别名**：
```bash
git remote -v
```

**添加远程仓库别名**：
```bash
git remote add 别名 远程地址
```

**案例**：
```bash
git remote add origin https://gitee.com/boncda/git-demo1.git
```
> 远程地址可在创建好的远程仓库页面中获取（HTTPS格式）。

**再次查看**：
```bash
git remote -v
# 输出示例：
# origin  https://gitee.com/boncda/git-demo1.git (fetch)
# origin  https://gitee.com/boncda/git-demo1.git (push)
```

---

#### 5.2 推送本地分支到远程仓库

**语法**：
```bash
git push 别名 分支名
```

**案例**：将本地 `master` 分支推送到远程仓库
```bash
git push origin master
```
> 💡 **首次推送时会提示输入码云的用户名和密码**（即Gitee的登录账号和密码）。

推送成功后，刷新远程仓库页面即可看到本地上传的内容。

---

#### 5.3 克隆远程仓库到本地

**语法**：
```bash
git clone 远程地址
```

**案例**：
```bash
git clone https://gitee.com/boncda/git-demo1.git
```

**`git clone` 的三个作用**：
1. **拉取代码**：将远程仓库的所有代码下载到本地。
2. **初始化本地仓库**：自动执行 `git init`，生成 `.git` 文件夹。
3. **创建别名**：默认将远程仓库地址命名为 `origin`（可通过 `git remote -v` 验证）。

#### 5.4 邀请加入团队（协作开发）

当需要多人协作时，仓库管理员可以邀请其他成员加入仓库，赋予推送权限。

**操作步骤**：

1. 进入远程仓库页面，点击 **"管理"** 选项卡。
2. 在左侧菜单中选择 **"仓库成员管理"** → **"邀请用户"**。
3. 选择添加方式（直接添加、链接邀请、二维码邀请等）。
4. 直接添加时，输入对方的码云用户名或邮箱，搜索并选中。
5. 指定权限（如"管理员"或"开发者"），点击确认提交。
6. 被邀请的用户接受邀请后，即可拥有相应的仓库操作权限。

**测试效果**：

- 被邀请的成员登录自己的码云账号，修改仓库文件并提交（`git push`）。
- 管理员或其他成员拉取（`git pull`）后，可以看到更新后的内容。

---

#### 5.5 拉取远程库内容

```bash
git pull 远程库地址别名 远程分支名
```

**案例**：
```bash
git pull origin master
```

> **`git pull` 的作用**：将远程仓库指定分支的最新内容拉取下来，并**自动与当前本地分支进行合并**。相当于 `git fetch` + `git merge` 的组合。

### 6. SSH免密登录（了解）

除了HTTPS协议，Git也支持通过SSH协议访问远程仓库，配置后可以免去每次输入密码的麻烦。

##### 6.1 生成SSH密钥对

```bash
ssh-keygen -t ed25519 -C "你的邮箱@example.com"
```
按提示连续敲三次回车，使用默认配置即可。

##### 6.2 查看公钥内容

```bash
cat ~/.ssh/id_ed25519.pub
```
复制输出的公钥字符串（以 `ssh-ed25519` 开头）。

##### 6.3 将公钥添加到Gitee

进入仓库主页 → **「管理」** → **「部署公钥管理」** → **「添加部署公钥」** → 粘贴公钥内容 → 保存。

##### 6.4 验证SSH连接

```bash
ssh -T git@gitee.com
```
首次连接需要确认主机指纹，输入 `yes`。若返回类似如下内容，表示配置成功：
```
Hi XXX! You've successfully authenticated, but Gitee.com does not provide shell access.
```

##### 6.5 使用SSH协议操作仓库

配置完成后，操作远程仓库时使用SSH地址（如 `git@gitee.com:用户名/仓库名.git`）即可免密推送和拉取。

```bash
# 示例：使用SSH地址添加远程别名
git remote add origin_ssh git@gitee.com:boncda/git-demo1.git
# 推送时无需输入密码
git push origin_ssh master
```

---

#### 补充说明

- HTTPS和SSH两种协议**都可以正常使用**，SSH免密更便捷，HTTPS则无需额外配置，适合临时使用。
- 如果仓库是私有仓库，只有被授权的成员才能推送或拉取代码。

## 第6章 IDEA集成Git（本地库)

### 1. 配置Git忽略文件

#### 1.1 哪些文件需要忽略

在使用IDEA等IDE开发时，以下类型的文件**不应该**纳入版本控制：

| 类型                      | 说明                                           |
| ------------------------- | ---------------------------------------------- |
| **IDEA特定文件**          | 如 `.idea` 目录、`*.iml` 文件等，与IDE配置相关 |
| **Maven工程的target目录** | 编译生成的字节码文件、打包产物等               |
| **日志文件**              | `*.log`                                        |
| **压缩包**                | `*.jar`、`*.war`、`*.zip`、`*.tar.gz` 等       |

#### 1.2 为什么要忽略它们

- 与项目的**实际功能无关**，不参与服务器部署运行。
- 不同开发者的IDE版本或配置可能不同，忽略这些文件可以**屏蔽IDE工具之间的差异**，避免不必要的冲突。

#### 1.3 如何配置忽略

**第一步：创建忽略规则文件**

1. 在用户家目录（`C:/Users/用户名/`）下创建一个文件，命名为 `git.ignore`（前缀名可自定义）。
2. 文件内容参考模版：

```
# Compiled class file
*.class

# Log file
*.log

# BlueJ files
*.ctxt

# Mobile Tools for Java (J2ME)
.mtj.tmp/

# Package Files #
*.jar
*.war
*.nar
*.ear
*.zip
*.tar.gz
*.rar

# virtual machine crash logs
hs_err_pid*

# IDE files
.classpath
.project
.settings
target
.idea
*.iml
```

**第二步：在 `.gitconfig` 文件中引用**

找到Windows用户家目录下的 `.gitconfig` 文件（如 `C:/Users/用户名/.gitconfig`），在末尾添加：

```
[core]
    excludesfile = C:/Users/用户名/git.ignore
```

> ⚠️ **注意**：路径中必须使用**正斜线（`/`）**，不要使用反斜线（`\`）。

配置完成后，Git在进行版本控制时会自动忽略这些文件中指定的内容。

---

### 2. 定位Git程序

在IDEA中配置Git可执行文件路径：

**操作路径**：`File` → `Settings` → `Version Control` → `Git`

在 **“Path to Git executable”** 中指定Git的安装路径（如 `C:\Program Files\Git\bin\git.exe`），点击 **Test** 按钮，若显示版本号则说明配置成功。

---

### 3. 初始化本地库

**操作路径**：

1. 打开项目，点击顶部菜单 `VCS` → `Create Git Repository`。
2. 在弹出的对话框中选择当前项目所在的目录。
3. 点击确定后，项目根目录下会生成 `.git` 文件夹，表示本地库初始化成功。

此时IDEA中文件名颜色会变化（红色表示未追踪）。

---

### 4. 添加到暂存区

**操作路径**：右键点击项目 → `Git` → `Add`

执行后，文件变为绿色，表示已添加到暂存区。

---

### 5. 提交到本地库

**操作路径**：右键点击项目 → `Git` → `Commit Directory...`

在弹出的窗口中：
- 选择要提交的文件。
- 填写提交日志信息（Commit Message）。
- 点击 **Commit** 按钮完成提交。

> 💡 也可以使用快捷键 `Ctrl + K`（Windows/Linux）快速调出提交窗口。

---

### 6. 切换版本（查看历史记录 & 版本穿梭）

**查看历史版本**：
1. 右键点击项目 → `Git` → `Show History`。
2. 在底部Version Control面板中可以看到所有提交记录，显示提交哈希、作者、时间、日志信息等。

**切换版本**：
1. 在历史记录列表中，右键点击目标版本。
2. 选择 `Checkout Revision`，即可切换到该版本。

切换后，项目文件会恢复到对应版本的状态。

---

### 7. 创建分支

**操作路径**：
1. 点击IDEA右下角的 `Git: master`（当前分支名）。
2. 在弹出的 `Git Branches` 窗口中，点击 `New Branch`。
3. 输入分支名称（如 `hot-fix`），点击确定。

创建成功后，会自动切换到新创建的分支，右下角的分支名会更新为 `hot-fix`。

---

### 8. 切换分支

**操作路径**：
1. 点击IDEA右下角的当前分支名（如 `hot-fix`）。
2. 在弹出的分支列表中，选择目标分支（如 `master`）。
3. 点击 `Checkout`，即可切换。

切换后，项目文件会变为对应分支的最新内容。

---

### 9. 合并分支

**前提**：先切换到需要接收合并的目标分支（如 `master`）。

**操作路径**：
1. 点击右下角当前分支名 → 选择要合并的分支（如 `hot-fix`）。
2. 点击 `Merge into Current`。

**结果**：
- 如果**没有冲突**，分支直接合并成功，代码自动提交到本地库，无需手动执行 `git commit`。
- 合并成功后会有提示。

---

### 10. 解决冲突

#### 10.1 冲突发生场景

当 `master` 分支和 `hot-fix` 分支**在同一个文件的同一位置**都进行了修改时，合并就会产生冲突。

#### 10.2 解决步骤

**第一步：执行合并**
在 `master` 分支上执行合并 `hot-fix` 分支，IDEA会弹出提示框，显示存在冲突的文件列表。

**第二步：手动合并**
1. 点击冲突提示框中的 **Merge** 按钮（或右键冲突文件 → `Git` → `Resolve Conflicts`）。
2. IDEA会打开冲突解决界面，左侧显示当前分支（master）的内容，右侧显示被合并分支（hot-fix）的内容，中间是结果区域。
3. **人为决定**保留哪些代码，通过点击箭头按钮或直接编辑中间区域来完成合并。
4. 点击 **Apply** 按钮完成合并。

**第三步：提交**
冲突解决后，IDEA会自动将合并结果提交到本地库（自动 commit），无需手动操作。

> ⚠️ 注意：如果在IDEA外使用命令行解决冲突，`git commit` 时不能带文件名，但IDEA中图形化操作会自动处理。
